# Orvanta — مراجعة المشروع وخطة التحسين

آخر تحديث: 2026-07-10

## الحكم التنفيذي

Orvanta تجاوز مرحلة النموذج الاستعراضي وأصبح يحتوي على عناصر نظام تشغيل أعمال حقيقية: مركز قرار، دورة أفكار واعتماد، مشاريع ومهام وKPIs، سجل تدقيق، Action Queue، أساس تعدد المستأجرين، Public API، Webhooks، PWA، وتطبيق iOS أولي.

المشكلة الرئيسية لم تعد نقص الخصائص؛ بل تشتت الهوية، تفاوت جودة الواجهات، الاعتماد الجزئي على بيئة demo/in-memory، وعدم اكتمال التشغيل الإنتاجي والتكاملات والمراقبة.

## التحسينات المنفذة في هذه الدفعة

| المحور | التحسين |

| الصفحة الرئيسية | شعار رئيسي واضح، واستخدام العلامة فقط داخل بطاقة النظام بدل تكرار الشعار الكامل |
| حالات النظام | إضافة شاشات Loading وError و404 تحمل هوية Orvanta |

## نقاط القوة الحالية

1. **دورة تشغيل قابلة للتتبع**

```txt
Idea → Feasibility → Approval → Project → Tasks → KPIs → Action Queue → Audit
```

2. **حوكمة أفضل من أغلب نماذج الوكلاء**

- مصفوفة صلاحيات.
- اعتماد بشري.
- Audit Log.
- منع بعض الأفعال من تجاوز مركز القرار.

3. **نطاق تقني متقدم**

- Next.js + TypeScript.
- Supabase persistence.
- OpenAI integration.
- Public API وWebhooks.
- PWA.
- أساس Multi-tenant.
- تطبيق SwiftUI أولي.

4. **فصل جيد نسبيًا بين الواجهة ومنطق الشركة**

وجود مجلدات `lib/company` و`app/api` ومكونات منفصلة أفضل من وضع منطق التنفيذ داخل الصفحات.

## الملاحظات ذات الأولوية

### P0 — يجب إغلاقها قبل اعتبار المشروع إنتاجيًا

#### 1. التحقق من البيئة الإنتاجية

يجب ألا يُعتبر النظام Production إذا كان أي من التالي غير مضبوط:

```env
AUTH_ENABLED=true
NEXT_PUBLIC_SUPABASE_URL=...
SUPABASE_SECRET_KEY=...
API_SECRET_KEY=...
OPENAI_API_KEY=...
```

الإجراء: اجعل `/api/health` بوابة نشر، وارفض الترويج إلى Production إذا كانت `productionReady=false`.

#### 2. المصادقة

المصادقة موجودة على مستوى API، لكن تجربة تسجيل الدخول والجلسة والصلاحيات في الواجهة ليست مكتملة بما يكفي.

الإجراء:

- شاشة Login رسمية.
- جلسة Supabase Auth محفوظة وآمنة.
- عدم استخدام Basic Auth للمستخدم النهائي.
- إظهار المستخدم والدور والقسم في الواجهة.
- اختبار صلاحيات كل API حساس.

#### 3. تعدد المستأجرين

الأساس موجود، لكن ختم `tenant_id` وحده لا يكفي للعزل الحقيقي.

الإجراء:

- إضافة `tenant_id` لكل الجداول الحساسة.
- RLS حسب tenant.
- عدم قبول tenant من العميل دون تحقق.
- اختبارات تمنع قراءة أو تعديل بيانات Tenant آخر.

#### 4. التكاملات الخارجية

بعض Actions تنتهي إلى `WAITING_INTEGRATION` بدل تنفيذ حقيقي.

الإجراء: اختيار 3 تكاملات فقط للمرحلة الأولى:

1. Gmail / Email
2. Google Sheets / Drive
3. WhatsApp Business أو Shopify بحسب العميل المستهدف

#### 5. البيانات المالية

يجب أن يبقى Ledger هو المصدر المالي الوحيد، وأي Dashboard مالي يقرأ منه لا من جداول موازية.

الإجراء: إضافة reconciliation tests وتقارير Budget vs Actual مبنية فقط على Ledger.

### P1 — جودة المنتج

#### 6. الواجهة مزدحمة وظيفيًا

المشروع يحتوي على عدد كبير من الأقسام واللوحات، ما يرفع العبء الإدراكي.

الإجراء المقترح:

- الرئيسية: قرار اليوم + 3 مؤشرات + ما يحتاج تدخل المالك.
- نقل الأقسام التفصيلية إلى قائمة ثانوية.
- إظهار “ما الذي تغير منذ آخر زيارة؟”.
- عدم عرض بطاقات لا تحتوي بيانات حقيقية.

#### 7. توحيد مكونات التصميم

لا يزال هناك تداخل بين `globals.css` و`unified-theme.css` و`brand.css` وملفات الشعار.

الإجراء:

- دمج الهوية في ملف Design Tokens واحد.
- إنشاء مكونات موحدة: `PageHeader`, `MetricCard`, `EmptyState`, `ErrorState`, `StatusBadge`.
- منع inline styles في الشاشات الجديدة.

#### 8. حالات الخطأ والتحميل

أضيفت حالات عامة في هذه الدفعة، لكن بعض المكونات ما زالت تستخدم `.catch(() => {})`.

الإجراء:

- API client موحد.
- timeouts وretry محدود.
- رسائل خطأ قابلة للفهم.
- correlation ID للأخطاء.

#### 9. الاختبارات

اختبارات منطق الأعمال موجودة، لكن لا توجد تغطية كافية للمسارات الكاملة.

الإجراء:

- E2E: Idea → Approval → Project → Action Queue.
- E2E: رفض الاعتماد.
- E2E: صلاحيات الأدوار.
- E2E: tenant isolation.
- Visual regression للجوال والكمبيوتر.

#### 10. المراقبة والتشخيص

Console وAudit Log لا يكفيان لمراقبة الإنتاج.

الإجراء:

- Sentry أو OpenTelemetry.
- قياس API latency/error rate.
- تتبع تكلفة واستدعاءات LLM.
- تنبيه عند فشل Action أو Webhook.

### P2 — التوسع

#### 11. تطبيق iOS

التطبيق الحالي بداية، وليس نسخة كاملة من النظام.

الإجراء:

- API client موحد مع الويب.
- Keychain وFace ID.
- Push Notifications للقرارات.
- Offline cache.
- Deep Links إلى القرار والمشروع.
- Feature parity تدريجيًا، لا نسخ كل شاشة دفعة واحدة.

#### 12. نموذج المنتج التجاري

النطاق واسع: تجارة، استثمار، تداول، CRM، حكومة، BI، مالية.

الإجراء: تثبيت عرض أساسي واحد:

> نظام تشغيل لصاحب منشأة صغيرة أو متوسطة في السعودية يحول القرارات إلى تنفيذ قابل للمتابعة.

ثم بيع الوحدات الأخرى كإضافات.

## خطة 30 / 60 / 90 يومًا

### أول 30 يومًا

- تفعيل Auth إنتاجي.
- تشغيل Supabase schema وRLS.
- فحص `/api/health` في النشر.
- إكمال Email + Sheets.
- إضافة E2E للمسار الأساسي.
- تنظيف CSS المتكرر.

### 31–60 يومًا

- Tenant isolation كامل.
- Observability.
- Action retries وidempotency.
- Dashboard مبسط للمالك.
- Push notifications للقرارات.

### 61–90 يومًا

- تكامل ثالث حقيقي.
- تطبيق iOS بخصائص القرار والتنبيهات.
- Billing وخطط اشتراك.
- Onboarding للشركات.
- قياس activation وtime-to-first-value.

## الخلاصة

المرحلة التالية ليست إضافة أقسام جديدة. الأولوية هي:

```txt
Production Security
→ Real Integrations
→ Reliable Execution
→ Simpler Owner UX
→ Observability
→ iOS Decision Experience
```

أي تطوير خارج هذا التسلسل سيزيد التعقيد قبل تثبيت القيمة الأساسية.
