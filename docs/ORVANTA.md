# Orvanta — نظام تشغيل الشركات بالذكاء الاصطناعي

**Mission:** Create the most advanced AI-powered company operating system in the world.

Orvanta هي المنصة؛ و«النجمة الذهبية» أول شركة تعمل عليها. الفكرة الجوهرية:
شركة كاملة من وكلاء ذكاء اصطناعي تدير دورة الاستثمار من الفكرة إلى القياس،
والمالك يحكمها من مركز قرار واحد تحت مصفوفة صلاحيات صارمة.

---

## 1) تحليل المعايير العالمية — وما طبّقناه منها

| الشركة | المبدأ الذي تتفوق به | تطبيقه في Orvanta |
|---|---|---|
| **Apple** | التصميم يخدم الوضوح؛ تفصيلة واحدة مصقولة أفضل من عشر مزايا | هوية ink-navy/gold موحّدة على CSS variables، بنتو-جريد هادئ، RTL أصيل بخط Cairo، ولوحة أوامر ⌘K تصل لأي شيء بضغطة |
| **Amazon** | التميّز التشغيلي يُقاس: كل قائمة انتظار لها عمر وSLA ومالك | كل قرار معلّق يحمل عمره (`ageHours`)، ويُعلَّم **متأخر عن SLA** بعد 24 ساعة؛ المؤشر يظهر في مركز القرار ولوحة الأوامر |
| **Google** | البيانات مصدر الحقيقة والتعلّم حلقة مغلقة | Supabase مصدر ديمومة لكل وحدة (كتابة فورية + hydrate)، وحلقة تعلّم أسبوعية تعاير عتبة الثقة من قرارات المالك الفعلية |
| **Stripe** | الشفافية التشغيلية منتَج: صفحة حالة يثق بها العميل | صفحة `/status` حيّة: كل تكامل وحالته، عدّادات جداول القاعدة، وفحوصات جاهزية الإنتاج من `/api/health` |
| **Notion** | البساطة: قدرة هائلة خلف واجهة لا تحتاج شرحاً | مركز قرار واحد لكل شيء، فكرة يومية تلقائية، تدهور آمن لكل ميزة بدون مفاتيحها (النظام لا «يتعطل» أبداً) |
| **OpenAI** | الوكيل نظام: أدوار محددة + حوكمة + أدلة | 10 وكلاء بأدوار ورتب، دراسات ثلاثية لكل فكرة بعقد أدلة (confidence/assumptions/evidence)، وLLM حقيقي عند توفر المفتاح |

## 2) البنية

```txt
Idea → Study (3 وكلاء) → Aggregate → Approval (مصفوفة T0–T3)
     → Project + Tasks + KPIs → Action Queue → Ledger/ZATCA → Audit → Learning
```

- **الديمومة:** كل وحدة في الذاكرة كنسخة عمل سريعة + write-through إلى Supabase
  و`hydrateOnce` عند أول قراءة لكل نسخة خادم (`lib/supabase.ts`).
- **الحوكمة:** `AUTHORITY_MATRIX` (T0≤5k ذاتي، T1≤25k CEO، T2/T3 مالك+جدوى)
  مفروضة في الـ API لا الواجهة فقط، وكل توقيع يُسجَّل في `audit_log` غير القابل للتعديل.
- **الأمان:** كتابة الخادم عبر service-role فقط، RLS مفعّل بلا سياسات عامة،
  ومسارات التنفيذ الحسّاسة خلف بوابة مالك.

## 3) لوحة القياس (ما يجعلنا «الأفضل» قابلاً للإثبات)

| المؤشر | الأداة |
|---|---|
| زمن اتخاذ القرار (decision latency) | `oldestPendingHours` + `stale` في `/api/inbox` |
| صحة التكاملات لحظياً | `/status` + `/api/health` (productionReady) |
| دقة توصيات الوكلاء | `agentAccuracy` في حلقة التعلّم |
| سلامة المحاسبة | ميزان مراجعة متوازن دائماً + فواتير ZATCA |

## 4) الخارطة — الحالة

1. ✅ **Realtime-lite:** مؤشر تغييرات خفيف (`/api/company/feed` ~100 بايت) تستطلعه
   الواجهة وتعيد الجلب فقط عند تغيّر البصمة — مركز القرار وشارة القائمة حيّان
   (`lib/useLiveRefresh.ts`). الترقية إلى Supabase Realtime تستبدل الخُطّاف فقط.
2. ✅ **Multi-tenant (الأساس):** كل صف يُختم بـ `tenant_id` والقراءة تُفلتر عليه
   عند `ORVANTA_MULTI_TENANT=true` بعد تشغيل `docs/supabase-multitenant.sql`
   (`lib/tenant.ts`). العزل الكامل بسياسات RLS لكل مستأجر هو التالي.
3. ✅ **Public API v1 + Webhooks:** `/api/public/v1/{status,ideas,decisions}`
   بمفتاح Bearer، وأحداث صادرة موقّعة HMAC — التفاصيل في `docs/PUBLIC_API.md`.
4. ✅ **PWA:** manifest + service worker (تثبيت على الجوال، فتح دون اتصال)
   وتنبيهات محلية عند تجاوز قرارات معلّقة للـ SLA (`components/PwaRegister.tsx`).
   إشعارات Push من الخادم (VAPID) هي الخطوة التالية.
5. ✅ **وكلاء قابلون للتخصيص:** المالك يعيد تسمية أي وكيل أو يوقفه من `/company`
   (زر ✏️) — البنية (الرتب/الصلاحيات/التبعية) تبقى محكومة غير قابلة للتعديل،
   وكل تخصيص يُسجَّل في سجل التدقيق (`lib/company/agentOverrides.ts`).

## 5) ما بعد الخارطة

- Supabase Realtime كناقل بديل للـ feed cursor.
- عزل RLS لكل مستأجر + إدارة مستأجرين من واجهة المنصة.
- Push notifications من الخادم (VAPID) للجوال المقفل.
- مفاتيح API متعددة بنطاقات (scopes) و rate-limits لكل مفتاح.
