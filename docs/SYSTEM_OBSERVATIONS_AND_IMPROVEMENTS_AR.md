# ملاحظات النظام وسبل التطوير — Orvanta

آخر تحديث: 2026-07-12

هذا المستند يجمع كل الملاحظات على النظام الحالي، مبنية على فحص فعلي للكود
(81 مسار API، 25 صفحة، 29 ملف اختبار، 23 ملف SQL، طبقة المصادقة والتخزين
والكرونات). كل نقطة تُعرض بصيغة: **النظام الحالي ← سبل التطوير**.

## أولاً: نقاط القوة (موجودة فعلاً ويجب الحفاظ عليها)

1. دورة تشغيل كاملة وقابلة للتتبع: فكرة ← جدوى ← اعتماد ← مشروع ← مهام ← KPIs ← Action Queue ← سجل تدقيق.
2. حوكمة حقيقية: مصفوفة صلاحيات، اعتماد بشري، سجل تدقيق append-only، وقيود محاسبية مزدوجة مع فاتورة ZATCA.
3. تدهور آمن (graceful degradation): النظام يعمل بدون مفاتيح خارجية بوضع تجريبي بدل الانهيار.
4. طبقة AI متعددة المزودين (Gemini / Claude / OpenAI) مع تبديل تلقائي عند الفشل.
5. تغطية اختبارات وحدات جيدة (29 ملف اختبار) وتوثيق داخلي غني في `docs/`.

## ثانياً: جدول الملاحظات — النظام الحالي مقابل سبل التطوير

### أ. الأمان والمصادقة

| # | النظام الحالي | سبل التطوير |
|---|---|---|
| 1 | الوضع الشخصي مفعّل افتراضياً: `isPersonalOwnerMode()` في `lib/auth.ts` يعيد `true` ما لم يُضبط `ORVANTA_PERSONAL_MODE=false`، فأي طلب بلا توثيق يحصل تلقائياً على صلاحية OWNER كاملة. | قلب الافتراضي: الوضع الشخصي لا يعمل إلا بتفعيل صريح، وإضافة حارس على مستوى `middleware.ts` (تحقق من جلسة أو مفتاح) بدل الاعتماد على كل route منفرداً. |
| 2 | معرّف المستأجر يُقرأ من ترويسة `x-orvanta-tenant-id` القادمة من العميل دون تحقق من ملكيته. | ربط المستأجر بهوية المستخدم الموثقة (من token/جلسة) وعدم قبول tenant من العميل إطلاقاً؛ إضافة اختبارات عزل تمنع قراءة بيانات مستأجر آخر. |
| 3 | RLS في Supabase غير مفروض بالكامل؛ الاعتماد الأساسي على مفتاح service role من الخادم. | تفعيل RLS بسياسات per-tenant على كل الجداول الحساسة (ledger, approvals, audit_log...) كطبقة دفاع ثانية حتى مع service role. |
| 4 | التوثيق متناقض مع الكود: README يوصي بـ `AUTH_ENABLED=true` بينما `middleware.ts` يعلن أن النسخة شخصية والمصادقة التجارية معطلة. | مواءمة الوثائق مع الواقع: وثيقة واحدة تحدد "وضع التشغيل" الحالي (شخصي أم تجاري) وما المتغيرات المطلوبة لكل وضع. |

### ب. البيانات والتخزين

| # | النظام الحالي | سبل التطوير |
|---|---|---|
| 5 | نمط الكتابة الأساسي `persist()` غير حاجب (fire-and-forget): الذاكرة أولاً ثم Supabase في الخلفية، مع سجل `failed_writes` للإعادة. النجاح يُبلَّغ للمستخدم قبل تأكيد الحفظ. | جعل كل المسارات المالية والاعتمادات تستخدم `persistCritical()` أو RPC معاملات (transaction) حصراً، وإبقاء `persist()` للبيانات غير الحرجة فقط؛ وجعل Supabase مصدر الحقيقة بدل الذاكرة. |
| 6 | 23 ملف SQL متناثرة بين `docs/` و `database/` بدون ترقيم أو أداة migrations، مع تعليمات "شغّل هذا ثم ذاك" يدوية. | اعتماد Supabase CLI migrations: ملفات مرقمة بتسلسل زمني في مجلد واحد `supabase/migrations/`، ومسار ترقية واحد قابل للتكرار من الصفر. |
| 7 | Rate limiting والكاش يعملان بذاكرة العملية (`Map` في `lib/rateLimit.ts` و `lib/cache.ts`) — على Vercel serverless كل نسخة لها ذاكرتها فلا يوجد حد فعلي مشترك. | نقلهما إلى مخزن مشترك: Upstash Redis أو Vercel KV، أو على الأقل توثيق أن الحد الحالي "لكل نسخة" وليس إجمالياً. |
| 8 | التعافي من `failed_writes` يتطلب تدخلاً؛ لا توجد آلية إعادة تلقائية. | إضافة cron لإعادة محاولة `failed_writes` المعلقة مع حد أقصى للمحاولات وتنبيه عند الفشل النهائي. |

### ج. البنية المعمارية والنطاق

| # | النظام الحالي | سبل التطوير |

| 9 | ازدواجية طبقات النطاق: `lib/company/*` و `lib/company-os/*` و `companyExecutionSystem.ts` و `businessBrain.ts` و `enterpriseSystems.ts` تتداخل في المسؤوليات (حوكمة، مالية، تنفيذ) بأكثر من تنفيذ. | اختيار طبقة واحدة معتمدة (المرشح: `lib/company-os` لأنه الأحدث والأكثر اختباراً)، وترحيل ما تبقى إليها ثم حذف المكرر تدريجياً. |

| 11 | وحدة التداول (`lib/trading/*`: سكالبينج، Alpaca، وسيط سعودي) تعيش داخل نفس النظام وبنفس مستوى الصلاحيات. | عزلها كوحدة مستقلة خلف علم تشغيل صريح، مع ضوابط مخاطر إلزامية (حدود يومية، kill switch، بيئة paper-trading افتراضياً) قبل أي ربط بأموال حقيقية. |
| 12 | لا يوجد API client موحد في الواجهة؛ كل صفحة تنادي fetch بأسلوبها، وبعض المواضع تبتلع الأخطاء. | إنشاء client واحد (timeout + retry محدود + رسائل خطأ مفهومة + correlation ID) تستخدمه كل الشاشات وتطبيق iOS معاً. |

### د. التشغيل والمراقبة

| # | النظام الحالي | سبل التطوير |
|---|---|---|
| 13 | المراقبة = `console.error` + سجل تدقيق فقط؛ لا Sentry ولا مقاييس latency/أخطاء ولا تتبع تكلفة LLM. | إدخال Sentry (أو OpenTelemetry) لأخطاء الخادم والواجهة، وقياس زمن الاستجابة ونسبة الأخطاء لكل API، وتسجيل تكلفة وتوكنز كل استدعاء LLM. |
| 14 | 8 كرونات يومية في `vercel.json` (الدورة، الأفكار، الرادار، التنبيهات، الملخص، التقارير، التعلم) وفشل أي منها صامت. | تسجيل نتيجة كل تشغيل cron في جدول `cron_runs` (نجاح/فشل/مدة) وإرسال تنبيه عند الفشل أو الغياب عن الجدول. |
| 15 | لا يوجد CI إطلاقاً — مجلد `.github/workflows` غير موجود رغم وجود `typecheck` و`test` و`build` جاهزة في `package.json`. | إضافة GitHub Actions يشغّل typecheck + test + build على كل push وPR، ومنع الدمج عند الفشل. |
| 16 | تاريخ Git غير منضبط: رسائل مثل "temp"، "stop"، "last accident"، "unexpected" — يصعّب التراجع والمراجعة. | اعتماد رسائل commit وصفية، والعمل عبر فروع وPRs مع حماية الفرع الرئيسي. |
